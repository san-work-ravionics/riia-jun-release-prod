CRISP-DM Methodology
ASML · Double DQN RL Trading System · From Raw Data to Production
1
CRISP-DM Phase 1
Business Understanding

RITA targets ASML Holding N.V. (ticker: ASML, Euronext Amsterdam), a global leader in semiconductor lithography equipment whose stock exhibits strong trending behaviour driven by semiconductor capex cycles. The business goal is to train a reinforcement learning agent that makes daily capital allocation decisions in ASML stock, achieving a Sharpe ratio above 1.0 and a maximum drawdown below 10% — constraints derived from real-world portfolio management tolerances. The investment decision is deliberately simplified to three discrete allocations — hold cash (0%), invest half (50%), or fully invest (100%) — so the model focuses entirely on timing entry and exit rather than optimising position size. Success is defined as consistent out-performance versus a passive buy-and-hold benchmark on a fully held-out backtest period that the model never saw during training or seed selection.

2
CRISP-DM Phase 2
Data Understanding

The ASML dataset covers daily OHLCV price history from 2019 through mid-2025 — approximately 1,500 trading sessions spanning bull runs, the 2022 semiconductor correction, and subsequent recovery cycles, making it rich in both trending and mean-reverting regimes. From the raw OHLCV feed, eleven technical indicators are derived: RSI(14) for momentum extremes, MACD(12/26/9) and its signal line for trend momentum, Bollinger Band %B for price position relative to volatility bands, 20-day rolling volatility as a regime indicator, and daily log returns as the primary risk signal. K-Means clustering with k=3 on the full feature matrix reveals three distinct market regimes — bullish trending, sideways consolidation, and bearish stress — each characterised by different RSI and volatility profiles that require different allocation strategies. The feature correlation matrix confirms that RSI and MACD carry complementary but non-redundant momentum signals, while 20-day volatility is largely orthogonal to both, making all three dimensions genuinely informative for the DDQN state vector.

3
CRISP-DM Phase 3
Data Preparation

The full dataset is partitioned chronologically into training (80%) and validation/backtest (20%), preserving strict temporal ordering so that no future data ever influences past decisions — this is the critical safeguard against look-ahead bias in any time-series machine learning system. Feature normalisation applies z-score standardisation (zero mean, unit variance) computed exclusively on the training window, with the same mean and standard deviation then applied forward to validation and backtest windows to simulate the live inference setting. The DDQN state vector is constructed as a 14-day rolling window over all 11 normalised indicators, giving the model a short-term memory of recent market momentum at each daily step — long enough to capture a trading week's context without introducing excessive lag. Data preparation runs as Step 2 of the 8-step RITA pipeline and outputs a NumPy float32 array with shape (timesteps, 14, 11) consumed directly by the Stable-Baselines3 custom environment wrapper.

4
CRISP-DM Phase 4
Modeling

The model is a Double Deep Q-Network (DDQN), implemented via Stable-Baselines3 DQN, with two hidden layers of 128 units each and ReLU activations — the double-network architecture decouples action selection from value estimation to reduce the overestimation bias inherent in vanilla DQN. The state space is the 11-dimensional normalised daily feature vector observed over a 14-day window; the action space has three discrete choices — hold cash (0%), invest half (50%), or fully invest (100%). The reward function is designed to balance return capture with capital preservation: r = daily_PnL − 0.3 × max(0, drawdown − 0.05), where the second term imposes a graduated penalty for any portfolio drawdown beyond 5%, directly encoding the 10% maximum drawdown business constraint into the learning signal. Training runs for 100,000 timesteps with experience replay (buffer size 10,000) and a separate target network updated every 1,000 steps; three random seeds are trained and the seed with the highest validation Sharpe is promoted, preventing cherry-picking on the backtest set.

5
CRISP-DM Phase 5
Evaluation

Each RITA model build is evaluated across three temporally disjoint windows: the in-sample training period, an out-of-sample validation window used only for seed selection, and a fully held-out backtest period that the model and the seed-selection process never saw — ensuring an unbiased estimate of real-world performance. The best seed is chosen by validation Sharpe ratio rather than backtest Sharpe to prevent the practitioner from overfitting the backtest through repeated runs. Four core metrics — Sharpe ratio, maximum drawdown, CAGR, and win rate — are evaluated against the business constraints from Phase 1; additionally, daily returns are compared to a passive buy-and-hold ASML strategy to confirm genuine alpha generation. SHAP attribution on the learned Q-values confirms RSI(14) and the MACD line as the dominant allocation signals, providing model interpretability alongside quantitative performance — a key CRISP-DM requirement before production promotion.

6
CRISP-DM Phase 6
Deployment

RITA runs as a Dockerised FastAPI application on AWS EC2 (ap-south-1 / Mumbai), reverse-proxied through Nginx over HTTPS with Google OAuth JWT authentication for all dashboard endpoints. The 8-step training pipeline — data validation, feature engineering, model design, training, period configuration, backtesting, results extraction, and API hot-swap — is triggered from the Build section of this dashboard and executes asynchronously, with real-time step progress streamed back to the UI via polling. Model weights (Stable-Baselines3 PyTorch .zip artefacts) are persisted to a Docker volume mounted at /app/data, surviving container restarts and rolling deployments without requiring retraining — a critical operational requirement given 100k-timestep training cycles. The DS Lab, RITA main dashboard, FnO options dashboard, and Claude Desktop MCP server all share the same FastAPI origin, meaning every CRISP-DM artefact — raw data stats, SHAP values, backtest results — is immediately accessible across the entire platform without data duplication.
